## Purpose

`speckit-testplan` converts a feature's checklist items into a structured, executable **manual test plan**. Each checklist item (`CHK###`) becomes one or more concrete test cases (`TC-###`) with numbered steps, `curl` examples where applicable, and a precise **Expected** outcome.

The output is written to `FEATURE_DIR/test-plan.md` (created or overwritten).

---

## User Input

```text
$ARGUMENTS
```

If a focus area is provided (e.g. "api", "auth"), restrict test case generation to checklists whose file name or category headings match that keyword. If empty, process all checklists.

---

## Execution Steps

### Step 1 — Locate the feature

Run `.specify/scripts/bash/check-prerequisites.sh --json` from repo root and parse:

- `FEATURE_DIR` — absolute path to the active feature folder (e.g. `specs/004-token-wallet-service`)
- `AVAILABLE_DOCS` — list of documents available in `FEATURE_DIR`

All subsequent paths must use `FEATURE_DIR` as the base.

### Step 2 — Read source documents

Read the following files if they exist (skip silently if absent):

1. `FEATURE_DIR/spec.md` — extract the feature name, FR-### functional requirements, and SC-### success criteria
2. `FEATURE_DIR/checklists/*.md` — all checklist files; parse every `- [ ]` and `- [x]` item by `CHK###` id, description, and category heading
3. `FEATURE_DIR/plan.md` — optional; extract architectural decisions that affect test prerequisites
4. `FEATURE_DIR/contracts/` — optional; extract API shapes referenced in CHK items (request/response bodies, status codes)

If `checklists/` does not exist or is empty, stop and tell the user: "No checklist files found in `FEATURE_DIR/checklists/`. Run `/speckit-checklist` first."

### Step 3 — Apply focus filter

If `$ARGUMENTS` is non-empty, keep only CHK items from checklist files whose name or category heading contains the argument (case-insensitive). Log which files are included/excluded.

### Step 4 — Generate test cases

Map each CHK item to one or more TC-### test cases using this logic:

**Numbering scheme**
- Group test cases by checklist file (each file = one section in the test plan)
- Within each file, group by category heading (e.g. `## Requirement Completeness`)
- Assign TC numbers sequentially across the full plan starting at TC-001
- Reserve 10-number blocks per category (TC-001–TC-009, TC-010–TC-019…) to allow future additions

**Per test case, write:**

```markdown
### TC-{NNN}: {Concise test name derived from CHK description}

> CHK{###} — {original CHK text, truncated to 120 chars}

**Prerequisites**: {any setup required — running app, auth token, seeded data, etc.}

**Steps**:
1. {First concrete action — include exact curl command if the CHK references an API}
2. {Second action}
3. …

**Expected**:
- {Measurable, binary outcome — HTTP status, JSON field value, database state, log line, etc.}
- {One bullet per distinct assertion}

**Notes**: {Optional — deviation from CHK scope, known gaps, deferred items}
```

**Mapping rules**:
- A CHK item that references a specific HTTP endpoint → include a `curl` command using `$PP` (PaymentProcessor), `$IDENTITY`, etc. as env var placeholders
- A CHK item about data integrity → include a `psql` command using `$PG_CONN`
- A CHK item about config/behavior → include steps to set config, restart, and observe
- A CHK item that is already `- [x]` (complete) → still generate the TC, but prefix the test name with `[VERIFIED]`
- A single CHK may produce multiple TCs if it covers multiple distinct scenarios

### Step 5 — Write the test plan

Write `FEATURE_DIR/test-plan.md` with this structure:

```markdown
# Manual Test Plan: {Feature Name}

> Generated by `/speckit-testplan` from {N} checklist items across {M} files.
> Feature: `{FEATURE_DIR}`
> Date: {ISO date}

---

## Prerequisites

| Requirement | How to verify |
|---|---|
| App running | `dotnet run --project src/eShop.AppHost/eShop.AppHost.csproj` |
| Relevant service healthy | Aspire dashboard → service shows green |
| Seeded data | Log line from Aspire dashboard on first startup |

**Environment variables used in curl examples**:
```bash
export PP="https://localhost:XXXX"          # target service URL
export IDENTITY="https://localhost:XXXX"    # Identity.API URL
export PG_CONN="postgresql://postgres:postgres@localhost:5432"
export TOKEN=""    # set by get-token.sh
```

---

## {Section: Checklist File Name (without .md)}

> Source: `checklists/{filename}.md` | {N} items → {M} test cases

### TC-{NNN}: …
…
```

If `test-plan.md` already exists, **overwrite it entirely**. Print a one-line diff summary: `Updated test-plan.md: {old_tc_count} → {new_tc_count} test cases`.

### Step 6 — Summary report

Print a table:

```
| Checklist File | CHK Items | TC Cases Generated |
|---|---|---|
| api.md         | 28        | 32                 |
| …              | …         | …                  |
| **Total**      | **N**     | **M**              |
```

Then print: "Run `/speckit-scripttest` to generate shell automation scripts for these test cases."
