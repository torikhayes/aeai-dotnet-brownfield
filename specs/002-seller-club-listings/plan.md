# Implementation Plan: Seller Club Listings

**Branch**: `002-seller-club-listings` | **Date**: 2026-07-13 | **Spec**: `/specs/002-seller-club-listings/spec.md`
**Input**: Feature specification from `/specs/002-seller-club-listings/spec.md`

## Summary

Extend `Catalog.API` to allow authenticated users to list their own golf clubs for sale. This requires adding `SellerId`, `Condition`, and `ManufactureYear` fields to `CatalogItem` via an EF Core migration, new seller-scoped endpoints (create listing, view own listings, view by seller, delete/deactivate), and photo evidence storage to satisfy the Trust & Safety constitutional requirement. All existing catalog endpoints and contracts are preserved.

## Technical Context

**Language/Version**: C# (.NET 10, pinned via `global.json`)
**Primary Dependencies**: ASP.NET Core minimal APIs, EF Core + Npgsql, .NET Aspire orchestration, Identity.API JWT authentication
**Storage**: PostgreSQL (`catalogdb`); photo files via existing static asset mechanism in Catalog API
**Testing**: xUnit v3 + Microsoft.Testing.Platform; existing `Catalog.FunctionalTests`
**Target Platform**: Linux containers orchestrated by Aspire; local development on macOS
**Project Type**: Distributed microservices — extending existing `Catalog.API` service
**Performance Goals**: Seller-scoped queries return in under 200ms; no regression on existing catalog endpoints
**Constraints**: No changes to existing endpoint contracts; `SellerId` set server-side only; `AvailableStock` always 1 for seller listings
**Scale/Scope**: Single EF Core migration; ~4 new endpoints; photo upload field required per Principle IV

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

### Pre-Phase 0 Gate Review

- **Principle I (Microservices Continuity)**: PASS. Changes are contained within `Catalog.API`; no new services; follows existing EF Core + Aspire patterns.
- **Principle II (Token Ledger Integrity)**: PASS (N/A). No token-affecting logic introduced.
- **Principle III (Attribute-Based Valuation)**: ⚠️ AMENDMENT NOTE. Spec originally used `LikeNew` as a `Condition` grade. This plan corrects the enum to align with the constitution's lookup table grades: **New / Excellent / Good / Fair**. The `LikeNew` value is replaced with `Excellent` throughout.
- **Principle IV (Trust & Safety)**: PASS with amendment. Spec was missing the photo upload requirement. This plan adds it: a listing submission **MUST** include at least one photo. `CatalogItem` gains a `PhotoUrls` field; the listing creation endpoint validates photo presence before accepting the submission.
- **Principle V (Risk-Based Testing)**: PASS. No token/trade financial code paths in this feature; standard PR-with-tests discipline applies.
- **Principle VI (Scope Boundary)**: PASS. No fulfillment or shipping logic introduced.

**Gate Result**: PASS (with two amendments to the spec — condition enum and photo requirement).

## Implementation Phases

### Phase 0 — Research
- Confirm how `sub` claim is extracted from JWT in existing endpoints (e.g., `OrderingApi.cs` pattern)
- Confirm EF Core migration workflow (how existing migrations were generated and applied via Aspire)
- Confirm existing image/photo serving mechanism in `Catalog.API`
- Confirm auth middleware setup in `Catalog.API/Program.cs`

### Phase 1 — Design Artifacts
- `data-model.md`: Document `CatalogItem` extensions and new `Condition` enum
- `contracts/`: Document 4 new endpoint contracts (POST create, GET by-seller, GET my-listings, DELETE/deactivate)
- `quickstart.md`: Step-by-step validation of seller listing flow

### Phase 2 — Tasks (generated by `/speckit-tasks`)

Key implementation work:
1. **EF Core migration** — add `SellerId`, `Condition` (enum: New/Excellent/Good/Fair), `ManufactureYear`, `PhotoUrls` to `CatalogItem`
2. **Endpoint: POST `/api/catalog/items`** — authenticated; sets `SellerId` from JWT `sub`; validates photo presence; sets `AvailableStock = 1`
3. **Endpoint: GET `/api/catalog/items/by-seller/{sellerId}`** — public; paginated
4. **Endpoint: GET `/api/catalog/items/my-listings`** — authenticated; returns caller's listings only
5. **Endpoint: DELETE/PATCH `/api/catalog/items/{id}`** — authenticated; ownership check (403 if not owner or admin)
6. **Tests** — functional tests for all 4 endpoints including auth enforcement and ownership rules
